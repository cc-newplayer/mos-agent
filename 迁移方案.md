# MOS 自动化测试 - 硬编码现状与迁移方案

## 一、当前系统结构

测试系统分两条线：

- 闭环测试线：loop_workflow/ + skills/batch_executor/
  - 从外部接收指令，输入到决策对话，监控执行，生成报告
  - 已经参数化，给 Excel/JSON 文件就能跑

- 操作用例线：executor/runner.py
  - 47条用例，覆盖账号管理、事件源、决策对话、执行日志等模块
  - 每种操作写死了完整的 UI 操作序列


## 二、硬编码清单

### 可以通过文件配置解决的（改配置不改代码）

这些是纯数据问题，抽到 Excel 或配置文件里就行：

1. 闭环测试指令（cases/loader.py 第12-33行）
   - 5条指令写死：油管转写、热点新闻、KOL跟评(elonmusk)、粉丝挖角、X投票
   - 解决：Excel 用例文件里加"指令"列，loader 直接读取，不再靠关键词匹配推断
   - 影响范围：仅影响"全量跑操作用例时顺便跑闭环"的场景
   - 风险：无。loop_workflow 和 batch_executor 已经支持外部传入指令

2. 定时任务表单数据（executor/runner.py 第1506-1523行）
   - 任务名"养号"、指令"用主号组账号执行养号闭环"、时间"13:00"写死
   - 解决：Excel 用例的"测试数据"列里写 JSON，比如 {"name":"养号","instruction":"...","time":"13:00"}
   - 风险：无。只是把固定值换成从用例读取

3. 导入测试文件（executor/runner.py 第212-232行）
   - 文件名 test_accounts.txt、内容 "test_user_1\ntest_user_2\n" 写死
   - 解决："测试数据"列里指定文件路径和内容，或者在 test_data/ 目录下准备好文件
   - 风险：无

4. 菜单操作类型（cases/loader.py 第36-45行）
   - 只识别"增长闭环"和"定时任务"
   - 解决：Excel 里加"操作类型"列，直接标注 growth_loop / schedule_task / closedloop 等
   - 风险：无。只是分类方式从"猜"变成"读"

### 需要改代码逻辑的（迁移新系统时必须处理）

这些是 UI 操作路径写死的问题：

5. 操作分派逻辑（executor/runner.py 第86-107行 + 第127-160行）
   - 用"描述里有没有某关键词"来决定调哪个方法
   - 比如描述里有"导入"就调 _test_import_account，有"删除"就调 _test_delete_account
   - 问题：新系统的用例描述措辞变了就匹配不上

6. UI 选择器写死（runner.py 各方法内部）
   - 按钮文字：'button:has-text("批量上传账号")'、'button:has-text("新建账号")'
   - 表单定位：'input[placeholder*="账号"]'
   - 对话框：'.ant-modal'、'.ant-drawer'
   - 表格结构：假设 tr/td 布局、假设有 checkbox 列
   - 问题：新系统如果换了 UI 框架或改了按钮文字，这些全部失效

7. 对话卡片匹配（executor/loop_handler.py 第68-76行）
   - 老版本用"引流号"文字匹配
   - loop_monitor.py 已改为用指令前缀匹配（已解决）
   - 但 loop_handler.py 还残留旧逻辑

8. 完成信号检测（executor/loop_handler.py 第169-216行）
   - "MarketingAI任务执行报告"、"规划阶段出错"、"Error code: 500" 等信号写死
   - 如果新系统的报告格式或错误提示变了需要更新


## 三、迁移方案

### 方案A：结构化操作手册 + 通用步骤执行器

思路：把每个测试用例的操作步骤写成结构化格式，执行器按步骤通用执行。

操作手册格式示例（JSON 或 YAML）：

```json
{
  "case_id": "CASE-003",
  "name": "导入账号",
  "steps": [
    {"action": "navigate", "url": "/accounts"},
    {"action": "click", "target": "批量上传账号", "match": "button_text"},
    {"action": "upload", "target": "input[type=file]", "file": "test_data/accounts.txt"},
    {"action": "wait", "target": "上传成功", "match": "page_text", "timeout": 5000},
    {"action": "assert", "condition": "visible", "target": "成功", "match": "message"}
  ]
}
```

通用执行器支持的 action 类型：
- navigate：导航到 URL
- click：点击元素（按文字/selector/角色匹配）
- fill：填写输入框
- upload：上传文件
- select：下拉选择
- wait：等待元素出现/消失
- assert：断言结果
- screenshot：截图

优点：
- 新系统来了改操作手册就行，不改代码
- 操作步骤可读性强，非开发也能维护
- 可以逐步迁移，新旧系统共存

缺点：
- 需要开发通用执行器（约 300-500 行代码）
- 页面结构复杂时 target 定位可能不够精确
- 需要有人写操作手册（不过比写代码简单）

适合场景：操作路径相对固定、页面结构规范


### 方案B：AI 实时解读页面 + 动态操作

思路：给 AI 一份操作手册（自然语言），AI 实时看页面快照，自己决定下一步怎么操作。

工作方式：
1. 用例文件里写自然语言步骤："进入账号管理页面，点击批量上传，选择文件上传"
2. AI 打开浏览器，截图/获取页面快照
3. AI 根据手册和当前页面状态决定下一步操作
4. 循环直到完成或失败

优点：
- 极其灵活，页面改了也能适应
- 操作手册用自然语言写就行
- 不需要精确的 selector

缺点：
- 慢（每一步都要 AI 推理）
- 不稳定（AI 可能判断错误）
- token 消耗大
- 调试困难

适合场景：探索性测试、页面频繁变动、一次性验证


### 建议：方案A 为主，方案B 做兜底

理由：
- 操作用例的路径是确定的（Excel里写了步骤），不需要AI实时判断
- 方案A 跑起来快、稳定、可复现
- 新系统来了只需要更新操作手册文件，不用改执行器代码
- 如果某些复杂场景方案A搞不定，再用方案B补充


## 四、迁移步骤（如果决定动手）

### 第一步：先解决"文件配置就能搞定"的部分

改动最小、风险最低：
1. Excel 用例文件加"指令"列和"操作类型"列
2. loader.py 读这两列，不再靠关键词推断
3. runner.py 里的固定值（任务名、文件名等）改为从 case["test_data"] 读取

预计工作量：半天。不影响现有功能，向后兼容。

### 第二步：闭环测试适配新系统

需要确认的：
- 新系统的决策对话入口在哪（URL 和 UI 路径）
- 新建对话的方式是否变了（目前是直接在输入框输入指令）
- 完成信号是否一样（"MarketingAI任务执行报告"）

改动点：
- loop_monitor.py 里的导航路径
- 输入指令的方式（如果变了）
- 其他监控逻辑大概率不用动

### 第三步：操作用例走方案A

1. 开发通用步骤执行器（step_executor.py）
2. 把现有 runner.py 里的操作逻辑翻译成操作手册文件
3. 新系统来了之后，根据新系统的页面重写操作手册
4. runner.py 改为读操作手册 → 调通用执行器

### 第四步：验证和切换

1. 新旧两套并行跑一段时间
2. 确认通过率不低于现有水平
3. 切换到新方案，旧代码归档


## 五、新系统迁移前的检查清单

到时候需要确认：
- [ ] 新系统 URL 和登录方式
- [ ] 各模块的页面路径（/accounts, /decision 等是否变了）
- [ ] UI 框架是否变了（现在是 Ant Design）
- [ ] 按钮文字是否变了（"批量上传账号"、"新建账号"等）
- [ ] 决策对话的新建入口
- [ ] 完成信号的格式
- [ ] 哪些操作用例在新系统里还有、哪些没了、哪些是新的
